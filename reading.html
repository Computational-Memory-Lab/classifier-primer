Where to go next

You now have the whole picture. This chapter is the practical part: what to read, how to run things, and what to be ready for.

Read these four

You do not need to read everything. Read these four and you will understand what the project is doing and why; the rest are reference, to open when you hit the specific thing they explain. Each card links to the copy on the lab's Google Drive, under CML documents / Papers.

The rest of the reading list, by what it explains

Rugg & Curran (2007)…“FN400” and “LPP” are new to you. Six pages, and the standard reference for the familiarity/recollection account.
Curran (2004)…you want to know why FA vs CR is interesting. The FN400 tracks subjective familiarity, so it fires on false alarms too.
Noh et al. (2014)…you want to see the closest methodological neighbour outside this lab.
Zhang et al. (2023)…you want univariate and multivariate approaches side by side.
Tipping (2001)…you touch the RVM. This is the paper the vendored toolbox implements.
Chawla et al. (2002)…you touch SMOTE. Matters most for FA vs CR, where one class can be 14 trials against 211.
Kayser & Tenke (2006); Perrin et al. (1989)…you touch the Laplacian/CSD branch. Perrin is four pages.
Delorme & Makeig (2004)…you need EEGLAB, which did the original preprocessing back in 2010.
Rouder et al. (2009); Wetzels & Wagenmakers (2012)…you meet a Bayes Factor in the manuscript.
Friendly et al. (1982)…you need to look something up about the stimulus words. Do not read it.

Getting it running

Three things to know before you open MATLAB, in order of how much time they will cost you if you do not.

StepDetail
1. Run setup_paths Once per MATLAB session, before anything else. The pipeline resolves its helpers by bare function name across sibling folders, so nothing works without it. Undefined function 'run_lda_auc' means you skipped it.
2. Point it at your data Each stage reads a folder and writes another; the coupling between stages is the .mat files on disk, not function calls. Check the settings at the top of whichever stage you are running before you run it — especially the DATA_SOURCE switch, which selects which comparison arm you are asking for. There is no single driver that runs the pipeline end to end; each stage is opened and run by hand.
3. Get the data It is not in the repository — about 8.7 GB on the lab Google Drive at CML documents/Classifier Project/data. Drive keeps the files as cloud stubs that report their full size while occupying no disk, so force a real download before reading them; a partially-downloaded .mat fails in confusing ways.
README.md has the full pipeline diagram, and the repository documents where every data file came from and how it was checked.
Before you conclude something is broken, check whether someone already knew. This code came together over several years and several people, so a few things are missing and a few are broken. That is context, not the point of the project. MISSING_FILES.md says what is absent, WORKLOG.md says what has already been tried and ruled out, and README.md says how the pipeline actually works. Checking those first will save you from re-deriving an answer someone already wrote down.

Two local customs

RuleWhy
Never delete files Anything that looks redundant goes to Potentially Delete/ with a written rationale. A human makes the final call. In a project reassembled from lost laptops, the cost of deleting something that mattered is much higher than the cost of keeping clutter.
Redundancy is often deliberate The raw/filtered/laplacian switches, the fixed-bin extractor sitting next to the moving-bin one, the “Alternative” pipeline — those are comparison arms, and chapter 6 is what they are for. Tidying them into a single path would delete the experiment.

There is a third custom worth adopting rather than just tolerating: write down what you ruled out. That is what WORKLOG.md is for. A sweep that found nothing is a real result — it stops the next person spending a week rediscovering the same nothing — but only if somebody records it.

And when you cannot verify something, say so rather than smoothing it over. Most of the value in this project's documentation comes from someone taking the trouble to write down what they were unsure about.

The short version of everything

People study words and then judge them old or new. Every trial gets one of four labels, and we record EEG from 256 electrodes throughout.

Averaging those trials shows a real difference between remembered and unremembered words, but an average cannot tell you about any single trial. So we reduce each trial to 280 numbers and train a classifier to predict its label, testing only on trials it has never seen.

It works — AUC around 0.58, reliably, across 67 people. Modestly. Preprocessing choices barely change it, the classifier barely changes it, but how you represent the signal changes it a lot.

And nobody yet knows why spectral representations do better. That is the question.

If something on this site is wrong, unclear, or has gone out of date, it is a static page in the same repository as the code — fix it and open a pull request.

Built from the lab's own data and results: the group numbers are recomputed from outputs/, and the EEG traces and the trainable classifier from the recordings on the Drive. Regenerate the derived files with python3 site/build_data.py.
