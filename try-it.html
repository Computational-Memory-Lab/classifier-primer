What the γ slider is doing

Linear discriminant analysis does not just ask which features differ between Hits and Misses. It also measures how the features move together — electrode 21 at 400 ms and electrode 21 at 440 ms overlap in time, so they rise and fall almost in step, and the model can exploit that to work out which of them is actually carrying the signal.

To do that it needs the correlation between every pair of features. With 280 features that is 280 × 279 ÷ 2 = 39,060 correlations, estimated from a few hundred trials. Most of them are being measured from far too little data, so their values are mostly noise — and the model, taking them at face value, builds elaborate weightings on relationships that are not really there.

γ (gamma) is how much you tell it to distrust those measurements. It slides between two extremes:

  • γ = 0 — trust every measured correlation completely. Maximum flexibility, maximum noise-fitting.
  • γ = 1 — ignore them all and pretend the features are unrelated. Maximum caution, and it throws away real structure along with the noise.
  • γ = 0.5 — what the pipeline uses: shrink every measured correlation halfway towards zero. Keep the ones strong enough to survive being halved, discount the rest.

It is the same trade as chapter 4's flexibility slider, applied to a different knob — and how much it matters depends on how many trials the task has to spare.

Old vs New uses all 450 trials, so a training fold has about 405 of them against 280 features. That is just enough to estimate the whole correlation matrix, and dragging γ to 0 costs surprisingly little. Hit vs Miss and FA vs CR use only the 225 old or 225 new trials, so a fold has around 200 — fewer trials than features. There the correlation matrix is singular: there is not enough data to pin down all 39,060 numbers, so infinitely many answers fit equally well and the model has no principled way to choose. Shrinkage is what makes the problem solvable at all, and on those tasks γ = 0 visibly hurts.

Try it: switch to Hit vs Miss and drag γ from 0.5 to 0.

Things worth trying

The controls are not decoration. Each one demonstrates something the lab has actually had to think about.

Try thisWhat you should see, and why
Test on the training trials AUC jumps from about 0.54 to about 0.73. Nothing improved — you are marking your own homework. That gap is chapter 4's overfitting, measured on real data, and it is why cross-validation is not optional.
Drag γ to 0 Turns the safety off. On Old vs New it barely matters — there are just enough trials to manage without it. On Hit vs Miss it costs real AUC, because that task has fewer trials than features. See below for what γ does. The pipeline fixes it at 0.5.
One electrode instead of ten AUC drops but usually stays above chance. No single electrode carries the signal — it is spread out, which is exactly the argument for a multivariate approach over a single ERP measurement.
The FN400 or LPP window only You have just reduced the classifier to roughly the classic ERP analysis. Compare it to all 28 bins to see what the extra time points buy you. The bins are 100 ms wide on a 40 ms grid, so they cannot land on 300–500 exactly — the nearest four cover 280–500 ms.
FA vs CR, SMOTE off With 14 false alarms against 211 rejections, the model learns to always say “rejection”. Watch the fold-to-fold spread explode — some folds contain barely any false alarms at all.
Run it twice unchanged The number moves. Folds are drawn randomly each run, and on one participant that is worth a few hundredths of AUC. Any single result here is noise around a true value you cannot see.

One participant is not a result

That last row is the important one, so it gets its own demo. Below, the exact same classifier on the exact same data is run over and over — the only thing that changes is which trials land in which fold.

Nothing changes between runs except the random fold split.
Each run is a legitimate, correctly cross-validated analysis of real data.
This is why the lab reports 67 participants, not one. A single participant's AUC wobbles by several hundredths for no reason other than the shuffle. Only when you average one score per person across dozens of people does the number get stable enough to defend — which is exactly what chapter 6 does.

How honest is this?

Honest enough to be checkable, which is the only useful kind. The feature extraction is a port of feature_label_moving_bin.m; the classifier mirrors run_lda_auc.m — regularized linear discriminant with γ = 0.5, stratified folds, SMOTE on the training fold only, AUC averaged across folds.

Rather than take that on trust, we handed this page's own feature matrix back to MATLAB and ran the lab's real run_lda_auc.m on it:

The only thing that differs between the button above and the real pipeline is which trials land in which fold, which is the wobble the previous section is about.

That check is a script, not a claim — it ships alongside this site, together with a browser-based counterpart that checks the classifier. If someone changes the MATLAB feature extractor and forgets this page, those two are what catch it. The site's own README says how to run them.
